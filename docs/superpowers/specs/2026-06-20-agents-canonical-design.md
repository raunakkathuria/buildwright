# Design: AGENTS.md as the single canonical instruction file

**Date:** 2026-06-20
**Status:** Approved (design); pending spec review

## Problem

Buildwright keeps two near-identical root instruction files:

- `CLAUDE.md` — hand-maintained source of truth (committed).
- `AGENTS.md` — generated from `CLAUDE.md` by `make sync`, gitignored.

The content is duplicated and the relationship is invisible (neither file
references the other). `AGENTS.md` is now the cross-tool standard (Codex,
OpenCode, Cursor, and Claude Code all read it). Buildwright also *distributes*
`CLAUDE.md` into every downstream project via three paths, so the duplication
propagates outward.

Separately, Codex currently receives only the 5 `bw-*` skills (via the
`make codex` symlink) — no project-level instructions — even though `AGENTS.md`
at repo root is Codex's native project-instructions convention.

## Goal

Make `AGENTS.md` the single canonical, committed, hand-maintained instruction
file. `CLAUDE.md` becomes a thin committed pointer stub. Apply this across the
**full distribution surface** so downstream projects are AGENTS-canonical too.
Close the Codex gap by documenting that Codex consumes `AGENTS.md`.

## Non-goals

- No change to `.buildwright/` → per-tool-dir generation (`.claude/`,
  `.opencode/`, `.cursor/`). Those are generated from `.buildwright/`
  subdirectories, never from the root instruction file, so they are unaffected.
- No change to the skills/Codex symlink flow itself (only its documentation).
- No new generated artifacts. Both root files become static and committed.

## Design

### 1. Root files (this repo)

- **`AGENTS.md`** ← receives the full current `CLAUDE.md` content, committed and
  hand-maintained. Two internal edits to the moved content:
  - **Project Structure** section: mark `AGENTS.md` as canonical (was
    "Generated by make sync"); show `CLAUDE.md` as the pointer stub.
  - **Git Rules**: flip the "do not commit ... `AGENTS.md`" line — `AGENTS.md`
    is now canonical and committed; `CLAUDE.md` is a committed stub.
- **`CLAUDE.md`** → ~4-line pointer stub:

  ```markdown
  # Buildwright

  Project instructions for all AI assistants live in **AGENTS.md** at the repo
  root. Read that file.

  (AGENTS.md is the single source of truth — Claude Code, Codex, OpenCode, and
  Cursor all consume it.)
  ```

### 2. `scripts/sync-agents.sh`

Delete Section 3 (`CLAUDE.md → AGENTS.md`): the `generate_agents_md` function,
its `--check` branch, and the line-329 summary echo. Also update the header
comment block (line ~15) that documents `AGENTS.md ← CLAUDE.md`. Sync no longer
touches either root instruction file.

### 3. `.gitignore`

Remove the `AGENTS.md` entry (lines 40–41). `AGENTS.md` is now tracked.
`CLAUDE.md` remains tracked.

### 4. `setup.sh`

Add a `curl` for `AGENTS.md` (canonical) alongside the existing `CLAUDE.md`
curl. Both are now static committed files in the repo, so both download cleanly.

### 5. CLI distribution (`cli/`)

- **`cli/templates/AGENTS.md`** — new "symlink placeholder" file containing
  `../../AGENTS.md` (matching the existing `CLAUDE.md` placeholder convention).
  `prepack.js` resolves it to a real copy at pack time; `init.js` copies every
  template entry into the target project, so new installs get both `AGENTS.md`
  (canonical) and the `CLAUDE.md` stub automatically. No `init.js` change
  needed.
- **`cli/src/commands/update.js`** — no backward-compat migration (per decision
  below). Replace the existing single-file `CLAUDE.md` "add if missing" block
  with a small loop over `['AGENTS.md', 'CLAUDE.md']` that copies each from the
  extracted release only when absent locally. Existing files are left untouched
  (treated as user-owned). No hash logic, no new abstraction — KISS/YAGNI/DRY.

### 6. Codex (`.codex/INSTALL.md`)

Add a section: when Codex CLI runs inside a Buildwright project, it
auto-discovers `./AGENTS.md` for project-level instructions (its native
convention). State clearly that Codex now gets **both** project steering
(`AGENTS.md`) and the 5 `bw-*` skills (`make codex` symlink).

### 7. Docs (`README.md`, `cli/README.md`)

Update the two lines in each that describe `AGENTS.md` as
"generated by make sync" / "regenerated by make sync". New wording: `AGENTS.md`
is the canonical project-instructions file; `CLAUDE.md` is a pointer stub to it;
neither is generated.

## Verification

- `make sync` runs clean and no longer mentions `AGENTS.md`.
- `make sync-check` passes (no longer flags `AGENTS.md` as out of sync).
- `AGENTS.md` is tracked by git; `CLAUDE.md` is the stub; both committed.
- `git status` shows no generated-file churn from the removed Section 3.
- CLI unit tests pass: `cd cli && npm test` (includes `update.test.js`).
- Pack smoke test: `make test-cli` then `buildwright init` in a scratch dir
  produces both `AGENTS.md` and the `CLAUDE.md` stub.
- `setup.sh` dry-read: both `curl` lines reference files that exist at repo root.

## Decisions

- **No backward compatibility.** Existing downstream projects keep whatever
  `CLAUDE.md` they have; `buildwright update` simply adds `AGENTS.md` if absent.
  We do not migrate or rewrite existing files. This keeps the change small and
  avoids hash-tracking machinery.

## Risks / edge cases

- **Claude Code reading the stub:** modern Claude Code reads `AGENTS.md`
  natively; the stub also explicitly instructs reading it. Instructions are
  delivered either way.
- **git `core.symlinks=false`:** the template placeholder convention already
  handles this (text file containing the target path); `AGENTS.md` placeholder
  follows the same pattern, so no new risk.
